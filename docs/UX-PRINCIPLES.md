# 美好 Home · 交互规范

> 本文是当前交互行为原则，不是视觉 token 或页面版式权威。当前视觉来源是 Figma `社区资源共享_小程序_0709_v2`，已确认观察见 [`figma-mini-0626-inventory.md`](./figma-mini-0626-inventory.md)，实现值以 `miniprogram/src/uni.scss` 和页面代码为准。已废弃的 [`VISUAL-TONE.md`](./VISUAL-TONE.md) 与 [`DESIGN-TOKENS.md`](./DESIGN-TOKENS.md) 仅供历史追溯。

## 1. 核心设计原则

### 速度 = 清晰

用户说豆瓣分组切换"清晰"，不是因为视觉上多好看，而是**切换后内容立刻变化，无延迟**。

**所有交互的第一优先级是响应快。** 如果做不到真的快，就做到**看起来快**（先渲染骨架，数据到了再替换）。

### 管理员优先

管理员/老师/群主是内容创作主力（20% 的人贡献 80% 的内容），社群活跃度取决于他们。

**管理员的竞争对手不是别的 App，是微信群里直接打字。** 如果在美好 Home 发一条信息比在微信群打字慢，管理员就不会用。

### 温暖但不啰嗦

文案要有人味（"还没有动态哦"），但不能拖沓（不加"亲爱的用户您好"）。

### 静默成功，显式失败

- **成功**：轻反馈（toast，1.5 秒自动消失）
- **失败**：重反馈（modal，必须用户手动关闭）
- **原因**：成功是预期结果，不需要打断用户；失败是意外，必须让用户知道并决定下一步。

## 2. 六大交互状态

每个页面在不同生命周期阶段，可能处于以下 6 种状态之一。开发时必须显式处理每种状态，不能留空白。

### Loading（加载中）

| 场景 | 实现 | 注意 |
|------|------|------|
| 首次进入页面 | `uni.showLoading({ title: '加载中' })` + 数据到了立刻 `hideLoading` | 如果 < 300ms 完成，用户几乎看不到 loading，这就对了 |
| 板块 tab 切换 | Feed 区域显示内联 loading 文字（不用全屏 loading） | **目标 < 300ms**，要快到用户感觉"即时" |
| 下拉刷新 | scroll-view 自带 refresher 动画 | 数据到了设 `refreshing = false` |
| 按钮提交 | 按钮文字变"发布中..." + disabled | 不用额外 loading 弹窗 |

**反例**：不要在切换板块时弹全屏 loading 蒙层 —— 这会打断用户的浏览节奏。

### Empty（空状态）

| 场景 | 文案 | CTA |
|------|------|-----|
| 社区 Feed 无帖子 | "还没有动态哦" | "发第一条 →"（跳 create 页） |
| 某板块无内容 | "该板块还没有内容" | 无（不强制引导发帖） |
| 未加入任何社区 | "还没有加入社区" | "去加入 →"（跳 onboarding） |
| 搜索无结果（未来） | "没有找到相关内容" | "换个关键词试试" |

**样式**：居中显示，文字用 `$hh-color-text-mute`，CTA 按钮用描边主色。垂直位置在页面 40% 处（不是正中间，偏上更自然）。

### Error（错误）

| 错误类型 | 反馈方式 | 文案 |
|---------|---------|------|
| 网络异常 | `uni.showToast({ icon: 'none' })` | "网络开小差了，稍后再试" |
| 业务失败（发帖失败等） | `uni.showModal({ title, content })` | 标题用操作名（"发布失败"），content 用具体原因 |
| 表单校验 | `uni.showToast({ icon: 'none' })` | 直接说缺什么（"请填写社区名称"），不要说"参数不合法" |
| 权限不足 | 不弹错误，走 Permission Guard 状态 | — |

**原则**：
- `showToast` 用于**可恢复的轻错误**（重试即可）
- `showModal` 用于**需要用户决策的重错误**（是否重试？是否放弃？）
- **永远不要在 toast 里放技术术语**（"Error 500"、"INVALID_PARAMS"）

### Success（成功）

| 操作 | 反馈 | 之后 |
|------|------|------|
| 发帖成功 | `uni.showToast({ title: '发布成功', icon: 'success' })` | `switchTab` 到首页 |
| 编辑保存 | `uni.showToast({ title: '已保存', icon: 'success' })` | 留在详情页 |
| 删除完成 | `uni.showToast({ title: '已删除', icon: 'success' })` | `navigateBack` |
| 加入社区 | `uni.showToast({ title: '加入成功', icon: 'success' })` | `reLaunch` 到首页 |
| 登出 | `uni.showToast({ title: '已登出', icon: 'none' })` | 留在 profile 页 |

**原则**：成功反馈 = toast（1.5 秒消失） + 自动跳转/留在原页。不要弹 modal 让用户点"确定"。

### Confirm（确认）

仅在**不可逆操作**前使用：

| 操作 | 弹窗标题 | 弹窗内容 | 确认按钮文字 |
|------|---------|---------|------------|
| 删除帖子 | "确认删除？" | "删除后无法恢复" | "删除" |
| 退出社区（未来） | "确认退出？" | "退出后需要重新申请加入" | "退出" |

**不需要 Confirm 的操作**：编辑、发帖、切换社区、切换板块 —— 这些都是可逆或低风险操作。

**实现**：`uni.showModal({ title, content, confirmText, cancelText: '取消' })`

### Permission Guard（权限守卫）

全页面级别的拦截，不是弹窗。

| 条件 | 页面显示 | CTA |
|------|---------|-----|
| 未登录 | "请先登录" + "登录后才能发布内容" | "去登录" → 跳 profile |
| 已登录但无社区 | "还没有加入社区" + "加入社区后才能发帖" | "去加入" → 跳 onboarding |
| 已登录但非当前社区成员 | "你还不是「XX」的成员" + "加入社区后才能发帖" | "加入社区" → 调 apply API |
| 申请审批中 | "你的加入申请正在审批中，请耐心等待" | 无按钮 |

**样式**：
- 居中布局，占满页面 60vh
- 标题用 `$hh-font-h3` + `$hh-color-text`
- 说明用 `$hh-font-body` + `$hh-color-text-mute`
- CTA 按钮用描边主色（`$hh-color-primary` 边框 + 文字）

## 3. 响应速度规范

| 操作 | 目标延迟 | 超时处理 | 参考 |
|------|---------|---------|------|
| 板块 tab 切换 | < 300ms | 显示内联 "加载中..." 文字 | 豆瓣分组切换感 |
| 发帖提交 | < 2s | 按钮 "发布中..." + disabled | — |
| 页面首次加载 | < 1s | 先渲染结构，数据到了填入 | — |
| 下拉刷新 | < 1s | refresher 动画自带 | — |
| 社区切换 | < 500ms | 切换面板先关闭，Feed 区域 loading | — |
| 详情页加载 | < 1s | 先显示标题占位 | — |

**关键实现**：
- 板块数据在首页 `onMounted` 时预加载，切换时从缓存读，只有首次需要网络请求
- 社区切换后立刻更新 UI（社区名、板块 tabs），Feed 数据异步加载
- 不要用 `await` 阻塞 UI 渲染 —— 先渲染，后填数据

## 4. 文案语气指南

### 核心原则

**像朋友发微信一样说话，不像银行发通知。**

美好 Home 的调性是"温暖、沉淀、有人味"。文案要配得上这个调性。

### 对照表

| ❌ 生硬 | ✅ 有人味 | 备注 |
|---------|----------|------|
| "操作成功" | "发布成功" | 说清楚成功了什么 |
| "暂无数据" | "还没有动态哦" | 加"哦"让语气柔和 |
| "请勿重复提交" | （按钮变灰 + 转圈） | 用视觉反馈替代文字警告 |
| "网络请求失败" | "网络开小差了，稍后再试" | 拟人化 |
| "参数不合法" | "请填写社区名称" | 说人话 |
| "未授权访问" | "登录后才能发布内容" | 告诉用户怎么解决，不要指责 |
| "Error: timeout" | "加载有点慢，再试一次吧" | 永远不露技术术语 |

### "哦"的使用规则

- 空态可以用"哦"：「还没有动态哦」「该板块还没有内容哦」
- 成功/失败不用"哦"：「发布成功」（不是"发布成功哦"）
- 确认弹窗不用"哦"：「确认删除？」（不是"确认删除哦？"）

## 5. 按钮反馈规范

### 三种按钮

| 类型 | 样式 | 用在哪 |
|------|------|--------|
| **主按钮** | `$hh-color-primary` 底 + 白字 | 发布、创建社区（每页最多 1 个） |
| **描边按钮** | `$hh-color-primary` 边框 + 文字，白底 | 去登录、去加入、次要操作 |
| **文字按钮** | 纯文字，无底色无边框 | 编辑、删除、取消 |

### 状态变化

| 状态 | 主按钮 | 描边按钮 | 文字按钮 |
|------|--------|---------|---------|
| 默认 | 木色橙底 + 白字 | 木色橙边 + 橙字 + 白底 | 语义色文字 |
| 按下 | 底色变 `$hh-color-primary-dark` | 底色变 `$hh-color-hover` | 颜色加深 |
| Loading | 文字变"XX中..." + disabled | 同上 | 文字变"XX中..." |
| 禁用 | `opacity: 0.4` | `opacity: 0.4` | 文字变 `$hh-color-text-mute` |

## 6. 跨页面规范

### 连击/并发防护

**所有写操作按钮必须用 `useBusyLock` 或 `useKeyedBusyLock`。**

前端是防重复提交的**唯一防线**（后端不保证去重）。没有 lock 的按钮 = 连点 5 次可能创建 5 条重复帖子。

### 页面跳转策略

| 方法 | 用在哪 | 效果 |
|------|--------|------|
| `uni.switchTab` | 底部 tab 之间切换（首页↔发帖↔我的） | 保留 tab 页面状态 |
| `uni.navigateTo` | 进入子页面（详情页、onboarding、createCommunity） | 保留来源页，可返回 |
| `uni.navigateBack` | 详情页返回列表 | 回到来源页 |
| `uni.reLaunch` | 重置全局状态（登出后重选社区） | 关闭所有页面，重新开始 |

### 返回行为

- 详情页左上角 `←` = `navigateBack`
- 发帖成功 = `switchTab` 到首页（不是 back，因为要刷新 Feed）
- 加入社区成功 = `reLaunch` 到首页（重置导航栈）
- 创建社区成功 = `navigateBack`（回到 onboarding）

## 7. 审计检查清单

每完成一个页面的交互调整后，用这个清单自检：

- [ ] **Loading**：首次加载有反馈？数据到了 loading 消失了？
- [ ] **Empty**：无数据时显示了友好空态？有 CTA 引导？
- [ ] **Error**：网络失败有 toast？业务失败有 modal？文案是人话？
- [ ] **Success**：写操作后有 toast？自动跳转/留在正确的页面？
- [ ] **Confirm**：不可逆操作有确认弹窗？可逆操作没有多余弹窗？
- [ ] **Permission**：未登录/非成员有全页面守卫？CTA 能跳到正确的页面？
- [ ] **按钮反馈**：按下有色变？loading 有文字变化？禁用有透明度？
- [ ] **连击防护**：所有写操作按钮用了 `useBusyLock`？
- [ ] **文案语气**：没有技术术语？像朋友说话？
- [ ] **响应速度**：板块切换 < 300ms？提交 < 2s？
