## 1. 产品定位（一句话）

> **美好 Home 是服务"社群"的助手**，帮社群完成两件事：
> 1. **信息沉淀**：把社群产生的有价值内容（出游路线、学习资料链接、活动经验等）系统化地留下来、可回查。
> 2. **实时同步**：拼车、活动报名、日程同步等轻度协作。

一句话总结：**社群的"时间胶囊 + 协作板"**。

---

## 2. 内在矛盾与解法

产品有一个**必须被明确化的内在矛盾**，否则 UI 决策会在两股力量之间摇摆。

| 力量 A：内容发现 | 力量 B：信息沉淀 |
|---|---|
| 小红书、抖音、即刻 | Notion、飞书文档、豆瓣小组 |
| 让用户**刷停**、沉浸 | 让用户**找到**、回溯 |
| 高密度瀑布流、图文优先 | 结构化列表、层级清晰、可筛选 |
| 强情绪 | 强结构 |

**解法（产品层）**：**气质模仿小红书（力量 A），形式保留工具属性（力量 B）**。
- "气质模仿"：温暖配色、圆润形状、友好文案、有人味的空态插图。
- "形式保留"：单栏卡片流而不是双列瀑布流、每条内容都带"板块 / 类型 / 时间"元信息、支持按板块筛选和搜索。
- **反面教材**：如果我们照搬小红书的双列瀑布流 + 图片为主，历史内容很快会沉到下面找不回来 —— 这违背了产品核心价值。

## 3. 调性关键词

产品给人的感受应该是 ——

| ✅ 是 | ❌ 不是 |
|------|--------|
| 温暖（像木质书架、老朋友家） | 冷淡（像银行 App、企业 SaaS） |
| 沉淀（有重量、可信赖） | 浮躁（抢眼的弹窗、红点、徽章） |
| 友好（文案像人说话） | 生硬（"操作成功"、"请勿重复提交"） |
| 有人味（手写感、自然角度） | 工业化（纯几何、网格感过强） |
| 轻结构（板块、标签清晰可辨） | 纯情绪化（只有图片和红心） |

**三词核心**：**温暖、沉淀、有人味**。

## 4. 视觉锚点

| 维度 | 锚点 |
|------|------|
| **语感** | 小红书（早期）的文案 + 圆润 + 暖色 |
| **气质** | 木质书架 / 纸质笔记本 / 豆瓣早期 |
| **结构** | 即刻的单栏 Feed + 飞书文档的板块层级 |

**主参考小程序**（建议用手机扫码体验 3 次以上）：
- 小红书（语感学习）
- 即刻（单栏 Feed 结构）
- 豆瓣小组（沉淀感 + 社群感）

## 5. 品牌色决策

### 主色 = `#E07A5F` · 木色橙

**为什么是它**：
- 属于"温暖色系"，符合小红书语感。
- 比小红书正红 `#FF2442` 更**沉稳、有重量**，契合"沉淀"的产品价值。
- 色度灵感：木质书架、宝丽来相框、傍晚厨房的灯光 —— 都是"把东西留下来"的场景色。
- 性别 / 年龄中性（比珊瑚粉更通用）。

### 颜色语义表

| 角色 | 色值 | 使用场景 |
|------|------|---------|
| Primary | `#E07A5F` | 主按钮、主要链接、选中态 |
| Primary Light | `#F5B9A8` | 主按钮 hover、Tag 底色 |
| Primary Dark | `#C25D43` | 主按钮 active、强调文字 |
| Success | `#52C41A` | 提交成功、已加入 |
| Warning | `#FAAD14` | 权限提示、非阻断性警告 |
| Danger | `#C0392B` | 删除、退出确认（暖红色，明确读作"红"，跟 primary 有清晰色相差） |
| Info | `#6E8CA0` | 温暖蓝灰，用于次要说明（不抢主色） |

### 中性色：**暖调微偏 + 纯白底**

- 背景主色是**纯白 `#FFFFFF`**（不走米白，为了商业产品的普适性）
- 次级底色 `#F7F6F3` 微微带暖意（跟 `#F5F5F5` 的区别：多了一点米黄 → 跟主色同温度）
- 边框 `#EBE3D8` 暖灰，避免冷灰跟主色冲突
- 弱文字 `#766F65` 暖灰（对比度 4.96:1，确保小号时间戳也能被阅读）
- 主文字 `#2C2416` 暖黑（比纯黑 `#000` 柔和，对比度 15.3:1 充足）

### 对比度声明（WCAG AA）

| 组合 | 对比度 | 达标？ | 用在哪 |
|------|--------|--------|--------|
| 主文字 `#2C2416` on 白底 | 15.3:1 | ✅ AA | 标题、正文 |
| 次文字 `#595550` on 白底 | 7.39:1 | ✅ AA | 副标题、说明 |
| 弱文字 `#766F65` on 白底 | 4.96:1 | ✅ AA | 时间戳、placeholder |
| Primary `#E07A5F` on 白底 | 2.95:1 | ❌ | **不用于正文链接**，改用 `$hh-color-primary-text` |
| Primary-dark `#C25D43` on 白底 | 4.25:1 | ✅ AA-lg | 正文链接、小字强调 |
| 白字 on Primary 按钮底 | 2.95:1 | ⚠️ 接受 | 大号按钮（行业惯例，见下方说明） |
| 白字 on Danger `#C0392B` | 5.44:1 | ✅ AA | 危险确认按钮 |

**已知 tradeoff：白字 on Primary 按钮**
- 温暖色系的 Primary 天生对比度偏低（Ant Design orange = 2.7:1，Tailwind = 更低），硬达标需要暗到 `#B85540`，温暖感会大幅流失。
- 小程序按钮通常 ≥28rpx 且高度 ≥88rpx，用户靠**位置 + 形状**识别，不纯依赖文字对比度。
- **守则**：Primary 色只能用于按钮/图标/Tag 等大号元素；正文中需要 Primary 色的文字链接，一律用 `$hh-color-primary-text`（= primary-dark `#C25D43`，4.25:1）。

## 6. 形式决策

### 信息密度：**中等**

**选择**：单栏卡片流（每屏 3–5 张卡片），而不是小红书式双列瀑布流。

**每张卡片标配元信息**（缺一不可）：
- 板块标签（如 "亲子"、"课堂"）
- 内容类型（动态 / 沉淀资料 / 报名）
- 标题 + 1–2 行摘要
- 作者 + 时间

**理由**：
- 元信息 = 沉淀和回查的钩子。纯图片瀑布流扔掉了所有钩子。
- 中等密度兼顾"浏览感"和"信息量"，一屏既不空也不挤。

### 圆角：**中等偏大**

- 卡片、主按钮：`16rpx`（小红书的温度感）
- 标签、小按钮：`8rpx`（保留轻量工具感）
- 大面板、底部抽屉：`24rpx`
- 头像、pill：`9999rpx`

**理由**：
- 圆角是**温度感最直接的表达器**。直角冷、圆角暖。
- `16rpx` 是"温而不腻"的甜区，大于它开始显得卡通。

### 字体：**系统字体，不引入自定义字体**

- 微信小程序 + H5 都用 `-apple-system, BlinkMacSystemFont, "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif`
- **原因**：
  - 包体积考虑，引入网络字体成本高
  - 苹方、鸿蒙黑、微软雅黑本身都契合"温暖感"（对比 Times New Roman 的冷硬）
  - 用户设备上的系统字体已经优化过可读性

## 7. 做与不做清单（决策速查）

### ✅ 做

- 主色只用木色橙一个品牌色，不加第二个撞色 accent
- 图标优先用**线性**图标（细线条），而不是面性图标（色块）—— 更契合"温和"感
- 动效保持**轻巧柔和**（`0.15–0.25s` 标准 easing）
- 空态、错误态都配**一句有人味的文案**（"还没人发动态哦" vs "暂无数据"）
- 所有交互按钮带**按下态视觉反馈**（颜色加深、轻微缩放）

### ❌ 不做

- 不用红绿灯式的高饱和色块做强提示（违背温度感）
- 不用粗黑字重 `900` 做堆砌视觉（违背沉淀感）
- 不用弹窗 / 震动 / 红点 做"刷存在"式设计
- 不用**纯灰**（`#CCCCCC` 这种）—— 所有灰色都要微微偏暖
- 不滥用阴影。只在需要"层"的概念时用（卡片、浮层），其他地方一律扁平

## 8. 示例：理想与反例

**理想状态**（Phase 4 完成后）：
- 打开首页：米白边的页面底色 + 纯白卡片 + 木色橙的"发布"FAB → 温度感立刻出来
- 点进详情：标题用暖黑、时间用暖灰、正文宽松行距 → 像翻一本纸质笔记
- 删除操作：木色橙系的 `danger` 确认按钮 + 文案"确认删除？删除后不可恢复" → 不吓人也不隐藏严重性

**当前反例**（Phase 0 前）：
- 颜色 `#333` + `#999` + `#fff` 三色灰阶 → 冷且没有品牌记忆
- 按钮直角 + 纯黑边框 → 工具感过强，跟"社群"气质冲突
- Toast 文案"操作成功" → 生硬

---

## 9. 变更流程

这份文档的任何修改需要满足以下之一：
1. 产品定位本身变了（不再是"社群助手"）
2. 用户调研发现目标人群对"温暖感"的反馈是负向的
3. 跟商业合作方签约导致主色必须调整

**不在这三项之列的修改一律拒绝** —— 否则设计系统会被每周的主观偏好摧毁。
