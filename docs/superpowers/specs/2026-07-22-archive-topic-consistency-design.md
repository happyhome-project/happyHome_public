# 沉淀区话题一致性修复设计

> **Point-in-time:** 本文件记录 2026-07-22 的修复设计与实施契约。
> **Current authority:** 以[文档权威索引](../../README.md)、当前代码、发布声明和测试为准。

日期：2026-07-22
状态：已确认，实施中

## 问题

沉淀区“全部”直接查询 `posts`，具体话题 Tab 查询 `archive_post_topics`。当前写入只追加话题链接和计数，删除、改话题与话题改名没有完整对账，导致事实数据与查询投影分离：后台数量累计历史帖子，帖子也可能只在“全部”出现。

话题改名还保留旧 `topicKey`，而发帖按新显示名生成新 `topicKey`。因此“家有小孩”改名为“教育成长”后，同一显示名会产生两个内部身份，Tab 与新帖子可能落在不同 key 上。

## 第一性原则

1. `posts` 是帖子状态、审核状态和所选话题的唯一事实源。
2. `archive_topics` 保存社区话题的稳定身份和显示配置；改名不改变稳定 `topicKey`。
3. `archive_post_topics` 是为了话题分页查询而建立的确定性投影，必须能够由 `posts` 和 `archive_topics` 幂等重建。
4. 后台“帖子数量”定义为当前属于该话题、`status=active` 且 `auditStatus=pass` 的帖子数，不是历史累计发布次数。
5. 显示名匹配已有话题时，发帖必须复用已有稳定身份，不能创建同名新身份。

## 方案

- 增加话题身份解析：优先按社区现有启用话题的规范化 `displayName` 匹配，再回退到规范化文本生成新 key。
- 将追加式同步改为单帖对账：写入当前应有链接，并把该帖不再需要的旧链接标记为 deleted；重试不会重复计数。
- 审核、编辑和删除路径都调用同一对账入口，避免各路径各自维护部分字段。
- 管理后台列表并行统计每个话题当前有效链接数，覆盖返回的 `recentPostCount`。
- 发布迁移合并同社区同显示名的重复话题，优先保留社区显式排序中的稳定 key；随后按事实帖重建全部链接。

## 验收

- “家有小孩”改名为“教育成长”后，新帖选择“教育成长”，链接仍使用稳定旧 key，并出现在该 Tab。
- 缺失链接可被对账恢复；改话题与删除会清理旧链接。
- 后台数量只统计当前 active/pass 帖子，删除和审核拒绝后立即减少。
- 历史重复“教育成长”身份由迁移合并，社区排序只保留一个入口。
