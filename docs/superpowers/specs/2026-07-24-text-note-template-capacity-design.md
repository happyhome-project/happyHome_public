## 用户目标

正文页应该充分利用每张底纸真正可读的区域。不能因为所有主题共用一个保守
文本框而留下大块无意义空白，也不能为了填满而缩字、越过装饰、截断原文或在
详情页显示与发布预览不同的分页。

## 第一性原则

- 背景装饰决定可读版心，因此容量属于主题，不属于全局常量。
- 文字能否放下同时受宽度、行数、手动换行、字符实际宽度和中文避头尾标点
  影响，不能只按 JavaScript 字符数量切割。
- 标题页与正文页继续分离。正文放不下时增加页面，不缩小 `16px / 24px`
  的正文排版。
- 发布预览和详情展示必须复用同一个 `TextNoteDeck → TextNoteCover` 契约。
- 保存的仍是标题、原始正文和主题；不保存派生分页，不新增 API 或数据迁移。

## 六套正文布局契约

全部坐标均以 `370 × 498px` 文字卡为基准。理论容量用于设计沟通；生产安全
容量比理论容量少一个视觉单位，用来吸收浏览器/微信中文避头尾标点和亚像素
舍入造成的末行抖动。

| 主题 | 名称 | 正文安全区 `x/y/w/h` | 字体 | 行高 | 理论中文容量 | 生产安全容量 |
|---|---|---:|---|---:|---:|---:|
| `paper` | 社区便签 | `32/48/306/384` | 系统黑体 | `16/24px` | `19 × 16 = 304` | `303` |
| `mint` | 邻里日常 | `32/52/306/384` | 系统黑体 | `16/24px` | `19 × 16 = 304` | `303` |
| `slate` | 今日记录 | `36/58/298/384` | 系统黑体 | `16/24px` | `18 × 16 = 288` | `287` |

## 分页契约

`getTextNoteBodyLayout(theme)` 返回主题规范化后的唯一布局配置。
`createTextNoteDeck` 必须把已规范化主题传给 `paginateTextNoteBody`。直接调用
分页函数而不传主题时回退到 `paper`。

每个候选页面必须同时满足：

1. 按主题每行视觉单位计算出的行数不超过 `maxLines`。
2. 页面全部可见字符的视觉单位之和不超过 `safeCapacity`。

视觉字符权重保持 Unicode 安全：

- 中文和全角字符为 `1`。
- 常规 ASCII 为约 `0.65`，窄字符更小。
- `W/M/@/#/%/&` 等宽 ASCII 为 `1`，不能把连续 `W` 或宽 URL 当成窄字符。
- 空白约 `0.3`。
- Emoji、修饰符、旗帜和 ZWJ 组合按一个不可拆分字素处理，权重 `1.6`。
- 手动换行立即结束当前行；空行真实消耗一行。

分页继续优先保留句末和段落边界，所有 `sourceBody` 拼接后必须精确等于
规范化原文。

## 渲染契约

`TextNoteCover` 在正文页读取同一布局配置，并把安全区换算为相对
`370 × 498` 卡片的百分比。六套正文统一使用 `32rpx / 48rpx`
（在 402px 设计画布约等于 `16px / 24px`），主题只改变安全区、字体族和
现有可读文字颜色。

正文保持 `overflow: hidden` 作为最后防线，但测试必须证明正常分页不会依赖
隐藏溢出。标题页和紧凑缩略图不应用正文布局。

## 验收

- 六套布局的安全区、理论容量和生产安全容量有纯函数测试。
- 每套主题在安全容量处为一页，多一个中文视觉单位时生成下一页。
- 中文、连续宽英文、长 URL、Emoji、组合音标、手动换行和空行不丢失、不重复、
  不拆坏 Unicode 字素。
- 发布预览切换主题后重新分页；详情页由持久化主题生成相同页面。
- `TextNoteCover` 的正文位置、字号、行高、字体和颜色与 H5 样板一致。
- H5 和微信构建通过；在 `390 × 844` 与 `402 × 917` 检查正文不溢出、
  不遮挡背景装饰、底部空白明显减少。
