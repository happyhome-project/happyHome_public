# 沉淀区用户可见层设计

日期：2026-07-14  
状态：已确认，可直接实施

## 目标

将 HappyHome 首页的沉淀区从“按板块浏览和发布”改为统一内容流：用户发布图文贴或纯文字贴，帖子以双列瀑布流展示，并通过 `#话题` Tabs 归类。板块概念只保留在实时协作中。

本功能还要把历史沉淀板块和帖子无损迁移到新语义，提供管理员话题配置，并保证首页浏览、话题筛选和智能搜索覆盖同一套沉淀内容。

## 第一性原则与硬边界

1. 帖子是沉淀区的主实体；话题只是可索引的归类，不拥有帖子。
2. `section.type === 'realtime'` 的板块继续存在，只服务实时协作，不进入沉淀区 Tabs。
3. `section.type === 'evergreen'` 的历史板块退出用户可见导航，其名称成为历史话题来源。
4. 新沉淀帖不保存 `sectionId`；历史帖子可保留 `sectionId`，仅用于兼容旧详情渲染。
5. 历史迁移不改变帖子 ID、作者、时间、互动数据或正文，不删除板块或帖子。
6. 话题筛选必须使用索引和稳定分页，不允许扫描全部帖子后在内存过滤。
7. 首页浏览、话题 Tabs 与智能搜索应覆盖同一套审核通过的沉淀内容。
8. 图文和文字发布器、瀑布流与发布入口采用小红书式的内容创作语法，但不复制其商标、品牌资源或文案。

## 用户界面

### 首页层级

首页保留当前社区抬头和搜索。进行中的实时协作使用一条轻量入口展示；没有实时内容时不占位。其下是可横向滚动的粘性话题栏和统一双列瀑布流。

话题栏固定以 `全部` 开头，后面最多展示七个话题。来源和优先级为：

1. 启用的历史沉淀板块名称，沿用原板块顺序；
2. 管理员手动配置的话题，按管理员顺序；
3. 按近期活跃度排序的其他话题，补齐剩余位置。

所有名称使用现有话题规范化规则去重。实时协作板块名称不生成话题，也不进入 Tabs。

### 双列瀑布流

所有沉淀帖使用同一套卡片层级：封面、标题、作者与点赞数。图文贴以首图为封面；纯文字贴使用现有 `TextNoteCover` 主题生成封面。没有可用图片的历史结构化帖子使用确定性的文字封面降级。

卡片在首次分配列后保持稳定，图片加载不能导致左右列重新洗牌。`全部` 和各话题都使用稳定游标分页。切换 Tab 时从第一页加载并显示骨架屏；加载下一页保留现有内容。空态显示“暂无相关内容”并提供发布入口；加载失败保留已加载内容并提供重试。

### 发布入口

点击底部 `+` 打开小红书式白色底部创作面板，使用浅灰图标底、黑色线性图标和短标签，只保留三个等权入口：

- `发图文`：直接进入无板块图文编辑器；
- `写文字`：直接进入无板块纯文字编辑器；
- `发起协作`：进入实时协作板块选择，再复用现有动态表单。

前两个编辑器都提供标题、正文、`#话题`、草稿和发布。发布成功返回首页 `全部`，新帖审核通过后出现在流中。只有第三条路径出现板块概念。

### 详情兼容

新沉淀帖使用图文或纯文字详情。迁移后的历史帖子带 `origin: 'legacy_section'`，继续通过保留的 `sectionId` 使用原详情模板；首页卡片统一不意味着破坏旧正文展示。

## 数据模型

### Post 扩展

新沉淀帖：

```ts
{
  area: 'archive',
  origin: 'native_archive',
  format: 'image_text' | 'text',
  topics: string[],
  // no sectionId
}
```

迁移后的历史沉淀帖：

```ts
{
  area: 'archive',
  origin: 'legacy_section',
  topics: string[], // 至少包含原板块名
  sectionId: string // 只供旧详情兼容
}
```

### `archive_topics`

社区级规范话题目录。每个 `communityId + topicKey` 唯一：

```ts
{
  communityId: string,
  topicKey: string,
  displayName: string,
  origins: Array<'legacy' | 'admin' | 'organic'>,
  enabled: boolean,
  legacyOrder?: number,
  adminOrder?: number,
  recentScore: number,
  recentPostCount: number,
  legacySectionId?: string,
  createdAt: string,
  updatedAt: string
}
```

历史来源优先于管理员来源，管理员来源优先于热门补位。同名话题只保留一个目录项，但可以同时具有多个 `origins`。

### `archive_post_topics`

帖子—话题索引投影：

```ts
{
  communityId: string,
  topicKey: string,
  postId: string,
  sortKey: string,
  createdAt: string,
  status: 'active' | 'deleted',
  auditStatus: 'pending' | 'pass' | 'reject'
}
```

文档 ID 使用帖子 ID 与规范话题键的确定性组合，使创建和迁移幂等。具体话题按 `communityId + topicKey + status + auditStatus + sortKey` 查询；`全部` 按 posts 现有沉淀索引查询。

## API 与数据流

### 首页初始加载

首页 bootstrap 返回实时协作数据，以及沉淀区的 Tabs、`全部` 首屏和下一页游标，避免初屏额外串行请求。后续分页或切换话题调用独立沉淀列表接口。

### Post API

- `listArchiveTabs({ communityId })`：返回 `全部` 和最多七个话题。
- `listArchive({ communityId, topicKey?, cursor?, limit? })`：返回统一卡片数据、`nextCursor` 和 `hasMore`。
- `createArchive(...)`：沿用已合入的无板块创建契约，并建立话题目录与索引投影。

审核通过时激活话题投影；删除、审核撤回或恢复时同步投影状态。投影是查询索引，不是第二份帖子正文。

### 智能搜索

正式 RAG 索引扩展到审核通过的 section-free 沉淀帖和迁移后的历史沉淀帖。索引文档允许缺少 `sectionId`，但必须带 `communityId`、`postId`、`area` 和话题。删除或审核撤回继续清理索引。实时协作内容维持现有规则。

## 管理端

管理员新增“沉淀区话题”设置页，按历史迁移话题和手动话题分组展示，支持：

- 启用或隐藏话题；
- 调整历史话题和手动话题各自顺序；
- 新增、重命名和删除手动话题；
- 查看话题来源与近期帖子数。

隐藏话题只影响 Tabs，不删除帖子上的话题。删除手动来源时，只移除 `admin` origin；如果该话题仍有历史或 organic 来源，则保留目录项。实时协作板块管理保持原样。

## 历史迁移

迁移只处理 `section.type === 'evergreen'`：

1. dry-run 输出社区、板块、帖子、预计话题关系和无法推断格式的统计；
2. 为每个历史板块幂等 upsert 话题目录；
3. 分批为帖子补充 `area`、`origin` 和原板块名称话题；
4. 建立帖子—话题索引；
5. 为搜索索引安排幂等重建任务；
6. 输出更新数、跳过数、失败数和可复核样本 ID。

迁移不自动归档或删除 sections。用户界面停止读取 evergreen sections 后，它们自然退出导航；实际数据清理必须另立任务并重新授权。

## 错误处理与降级

- 首页 `全部` 不依赖话题索引，即使投影暂时缺失仍可浏览。
- 具体话题索引失败时显示可重试错误，不退化为不相关内容。
- 单条历史帖子迁移失败不终止整批；记录错误并允许按 ID 重跑。
- 旧帖子无法生成图片卡时使用稳定文字封面，但详情仍使用旧模板。
- 发布成功但列表刷新失败时保留成功提示，并提供返回首页重试，不重复提交帖子。
- 热门分数允许短暂最终一致，但历史与管理员配置必须强一致地反映在 Tabs 返回值中。

## 测试与验收

### 自动测试

- Cloud：话题优先级、规范化去重、稳定游标、投影生命周期、RAG section-free 索引、迁移幂等与 legacy 详情兼容。
- 小程序：三入口路由、`全部` + 话题切换、瀑布流稳定分栏、图文/文字混排、空态、重试和分页。
- Admin Web：话题新增、隐藏、排序、来源合并和删除语义。
- 静态治理：新增集合和复合索引声明、release change manifest、部署范围与迁移动作声明。

### 真实闭环

在具备验证租约和安全 fixture 的环境中：创建临时社区及 evergreen/realtime 板块和帖子，执行迁移，验证首页 Tabs 与双瀑布流，发布图文和文字，按话题筛选和搜索，再发起实时协作，最后清理 fixture。若共享环境或 DevTools 不可用，必须明确报告未覆盖的真实路径，不能把静态检查冒充闭环验证。

## 实施顺序

1. 建立话题目录、投影、游标查询、迁移和搜索兼容；
2. 连续完成小程序用户可见层：首页 Tabs、统一双瀑布流、三入口发布和新详情路由；
3. 不等待人工确认，直接完成管理端话题配置、索引/release 声明及完整验证；
4. 独立代码审查后推送 PR，跟踪 exact HEAD 的 PR CI 和 Merge Queue 至终态。

## 非目标

- 不删除历史 sections 或旧帖子字段；
- 不重新设计实时协作板块和动态表单；
- 不在功能 worktree 部署生产、执行生产迁移或上传小程序；
- 不新增视频发布类型；
- 不把实时协作板块名称加入沉淀区话题 Tabs。
